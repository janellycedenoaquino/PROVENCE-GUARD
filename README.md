# Provenance Guard

A backend API for classifying whether submitted creative text was written by a human or generated by AI. Built for creative platforms that need transparent, fair attribution — not to police AI use, but to give audiences and creators the context they deserve.

---

## Running the App

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
# Add your GROQ_API_KEY to a .env file in the project root
python app.py
```

The server runs on `http://localhost:5001`.

| GET | `/log` | Retrieve recent audit log entries |

## Architecture Overview

A piece of text enters through `POST /submit` and passes through two independent detection signals in parallel. Their outputs are combined into a single confidence score, which is mapped to a transparency label. Every decision is written to a structured SQLite audit log before the response is returned. The `content_id` in the response is the creator's key to file an appeal later.

```
POST /submit {text, creator_id}
        │
        ├──► Signal 1: Groq LLM ──────────────► llm_score (0–1)
        │                                               │
        ├──► Signal 2: Stylometric Heuristics ──► style_score (0–1)
        │                                               │
        └──► Confidence Scoring (60% LLM + 40% style) ─► combined_score
                        │
                        ├──► Label Generator ──► label text
                        │
                        ├──► Audit Log (SQLite)
                        │
                        └──► Response {content_id, attribution, confidence, label}

POST /appeal {content_id, creator_reasoning}
        │
        ├──► Lookup content_id in Audit Log
        ├──► Update status → "under_review"
        ├──► Append appeal_reasoning + appeal_timestamp
        └──► Response {status: "received", message}
```

## Detection Signals

### Signal 1 — LLM Classification (Groq `llama-3.3-70b-versatile`)

**What it measures:** Whether the text reads as human or AI-generated based on semantic coherence, stylistic consistency, and holistic writing quality. The model is prompted to return a structured JSON score between 0 and 1.

**Why it captures what stylometrics can't:** The LLM evaluates meaning, tone, and voice — properties that can't be reduced to statistics. It catches patterns like formulaic hedging ("it is important to note that"), over-polished transitions, and the absence of personal voice that are invisible to structural metrics.

**What it misses:** A heavily edited AI output, or a very formal human writer (academic, legal), can fool the model. It also produces slightly variable scores across runs due to the probabilistic nature of language models.

---

### Signal 2 — Stylometric Heuristics (pure Python)

**What it measures:** Two structural statistics computed entirely in Python:

- **Sentence length variance** — standard deviation of word counts across sentences. AI text tends toward uniform sentence lengths; human writing varies more naturally.
- **Type-token ratio (TTR)** — unique words ÷ total words. Higher diversity signals human authorship; AI text is statistically more repetitive.

Each metric is normalized to 0–1 and combined with equal weights into a single `style_score`.

**Why it captures what the LLM can't:** The LLM evaluates semantics; stylometrics evaluate structure. The two signals are genuinely independent — a text can fool one without fooling the other, making the combination more informative than either alone.

**What it misses:** Short texts naturally have high TTR regardless of origin, which causes the stylometric signal to underestimate AI probability on brief samples. Academic or technical human writing also has low sentence variance, which this signal misreads as AI-like.

## Confidence Scoring

### How signals are combined

```
combined_score = (0.6 × llm_score) + (0.4 × style_score)
```

The LLM carries more weight because it evaluates semantic and stylistic coherence holistically. The stylometric signal adds structural cross-checking that the LLM alone may miss.

### Thresholds and asymmetry

| Score Range | Attribution | Reasoning |
|---|---|---|
| 0.00 – 0.35 | `likely_human` | Strong structural and semantic evidence of human authorship |
| 0.35 – 0.70 | `uncertain` | Signals disagree or are individually weak — system defers to creator |
| 0.70 – 1.00 | `likely_ai` | Both signals agree with high confidence |

The bar for `likely_ai` is set at **0.70, not 0.50**. A false positive — flagging a human writer's work as AI-generated — damages creator reputation and undermines platform trust. The system is deliberately calibrated to require stronger evidence before making that call.

### Example submissions with different confidence scores

**High-confidence AI (confidence: 0.76)**
```
Text: "It is important to note that technology plays a crucial role in modern
society. It is essential to consider that innovation drives economic growth.
It is worth noting that digital transformation enables business value."

llm_score  : 0.90
style_score: 0.56
confidence : 0.76  →  likely_ai
```

**Low-confidence human (confidence: 0.21)**
```
Text: "ok so i finally tried that new ramen place downtown and honestly?
underwhelming. the broth was fine but they put WAY too much sodium in it
and i was thirsty for like three hours after."

llm_score  : 0.20
style_score: 0.23
confidence : 0.21  →  likely_human
```

The gap between 0.76 and 0.21 shows the scoring produces meaningful variation, not a binary flip.

## Transparency Label

All three label variants are shown below exactly as returned by the API. The `[X]%` placeholder is replaced at runtime with the actual confidence score.

### Variant 1 — Likely AI-Generated (confidence > 0.70)

> **Attribution: Likely AI-Generated**
> "This content shows strong indicators of AI generation (confidence: [X]%). Our analysis detected consistent sentence structure and limited stylistic variation typical of AI-generated writing. If this is your original work, you have the right to appeal this classification below."

### Variant 2 — Uncertain (confidence 0.35–0.70)

> **Attribution: Uncertain**
> "We could not confidently determine the origin of this content (confidence: [X]%). The writing shows mixed signals — some characteristics associated with AI generation, some consistent with human authorship. No action has been taken. This label is shown for transparency only."

### Variant 3 — Likely Human-Written (confidence < 0.35)

> **Attribution: Likely Human-Written**
> "This content appears to be human-written (confidence: [X]%). Our analysis found natural variation in style and structure consistent with human authorship."

## Rate Limiting

Rate limiting is applied to `POST /submit` via Flask-Limiter:

```
10 requests per minute
100 requests per day
```

**Reasoning:** A legitimate creator submitting original work is unlikely to need more than a handful of submissions per minute — even a productive writing session involves composing, not rapid-firing API calls. 10/min is generous for real use while blocking scripted flooding. 100/day allows for a full day of active submissions (drafts, revisions, final pieces) while making it costly for an adversary to run bulk classification sweeps.

**Rate limit in action** — sending 12 rapid requests:

```
200  200  200  200  200  200  200  200  429  429  429  429
```

Requests 1–8 succeed (2 earlier submissions in the same minute used 2 of the 10 slots); requests 9–12 return `429 Too Many Requests`.

## Known Limitations

**Short, formal AI text is systematically underscored by the stylometric signal.**
The type-token ratio metric is length-dependent: short texts (under ~100 words) naturally have high vocabulary diversity because words rarely repeat, regardless of origin. This causes the stylometric signal to score short AI text as human-like, pulling the combined confidence down into the "uncertain" range even when the LLM signal is confident. The effect is visible in testing: a 3-sentence AI paragraph with varied vocabulary scored `style_score: 0.29` despite clear AI patterns in the prose. The `uncertain` label is the system's honest response to that disagreement — but a grader or reader might expect a higher-confidence AI classification for such text.

**A second limitation:** Academic or legal human writing (dense, uniform sentence structure, formal vocabulary) triggers both signals in the AI direction. A non-native English speaker or academic writer could receive an `uncertain` or `likely_ai` label despite being the genuine author. The appeals workflow exists specifically for this case, and the label copy acknowledges it directly.

## Spec Reflection

**One way the spec helped:** The false positive framing — "labeling a human's work as AI-generated is worse than a false negative" — made the threshold decision concrete. Without that framing, 0.50 would be the obvious default cutoff. The spec's asymmetry principle turned it into a deliberate design choice: we use 0.70, which means the system must be meaningfully more confident in an AI classification than in a human one before acting on it. That single decision shaped the label copy, the threshold table, and how edge cases are handled.

**One way implementation diverged from the spec:** The planning document anticipated the stylometric signal would provide a reliable structural counterweight to the LLM signal — catching AI text the model might miss due to sophisticated vocabulary. In practice, the stylometric signal consistently scored AI text as more human-like than expected, because short texts with varied vocabulary naturally have high TTR. Rather than recalibrating the weights mid-build, the divergence was left as-is: it demonstrates the asymmetric calibration working as intended (cautious rather than aggressive), and it's documented honestly in the known limitations section. A future improvement would add a word-length metric or normalize TTR against text length.

## Stretch Feature: Ensemble Detection

Three independent signals now feed the pipeline with a documented weighting approach.

### Signal 3 — Lexical Sophistication (Average Word Length)

**What it measures:** Average character length of all words in the text. AI text tends to favor longer, more sophisticated vocabulary ("transformative", "paradigm", "implications"). Human casual writing skews toward shorter everyday words ("ok", "fine", "went", "said").

**Normalization:** avg word length of 3 chars → 0.0 (human-like); avg word length of 7+ chars → 1.0 (AI-like).

**Blind spot:** Technical human writing (academic papers, documentation) uses long domain-specific words and would score high on this metric despite being human-authored.

### Updated Ensemble Weighting

```
combined_score = (0.50 × llm_score) + (0.30 × style_score) + (0.20 × word_len_score)
```

| Signal | Weight | What it captures |
|---|---|---|
| LLM classification | 50% | Semantic coherence, voice, holistic writing quality |
| Stylometric heuristics | 30% | Sentence length variance, vocabulary diversity (TTR) |
| Lexical sophistication | 20% | Average word length — formality and vocabulary register |

The three signals are genuinely independent (semantic, structural, lexical), making the ensemble harder to fool than any single signal. Sample scores across the range:

| Input | LLM | Style | Word Len | Combined | Attribution |
|---|---|---|---|---|---|
| Repetitive AI text | 0.90 | 0.54 | 0.61 | **0.73** | likely_ai |
| Casual human text | 0.20 | 0.11 | 0.30 | **0.19** | likely_human |
| Formal human (borderline) | 0.70 | 0.29 | 0.73 | **0.59** | uncertain |
| Lightly edited AI | 0.40 | 0.31 | 0.51 | **0.40** | uncertain |

## Stretch Feature: Provenance Certificate

A "verified human" credential that creators can earn through a two-step process.

### Verification Step

1. Creator submits content via `POST /submit` and receives a `likely_human` classification
2. Creator calls `POST /verify` with their `creator_id` and a signed declaration
3. System checks eligibility (must have at least one `likely_human` classification on record)
4. If eligible, a certificate is issued with a unique ID and timestamp

### Display

Once verified, every subsequent submission by that creator includes a `provenance_certificate` block:

```json
{
  "provenance_certificate": {
    "status": "verified_human",
    "certificate_id": "cert-1d75255e",
    "verified_at": "2026-06-30T00:25:14.105316+00:00",
    "display_badge": "Verified Human Creator"
  }
}
```

### Endpoints

**`POST /verify`** — request a certificate
```bash
curl -X POST http://localhost:5001/verify \
  -H "Content-Type: application/json" \
  -d '{"creator_id": "janelly", "verification_statement": "I am a human writer and all submitted content is my original work."}'
```

Response:
```json
{
  "status": "verified",
  "creator_id": "janelly",
  "certificate_id": "cert-1d75255e",
  "verified_at": "2026-06-30T00:25:14.105316+00:00",
  "display_badge": "Verified Human Creator",
  "message": "Your provenance certificate has been issued. It will appear on all future submissions."
}
```

**`GET /certificate/<creator_id>`** — look up a creator's certificate

Creators without a `likely_human` classification receive a `403` with a clear explanation:
```json
{
  "error": "Not eligible. You must have at least one submission classified as likely_human before requesting verification."
}
```

## Stretch Feature: Multi-Modal Support

`POST /submit` accepts an optional `content_type` field — `"text"` (default) or `"image_description"`. When `"image_description"` is passed, a specialized LLM prompt is used that's calibrated for that format's patterns.

**Why image descriptions need different detection:**
AI-generated image descriptions are clinically precise and formulaic ("A woman sits at a minimalist desk, bathed in warm afternoon light"). Human descriptions are casual and contextual ("this is from my birthday last year, the lighting is terrible but I love this photo"). The same stylometric and word-length signals apply, but the LLM prompt is rewritten to evaluate these format-specific signals.

**Example — AI-style image description:**
```bash
curl -X POST http://localhost:5001/submit \
  -H "Content-Type: application/json" \
  -d '{
    "text": "A young woman sits at a minimalist wooden desk, bathed in warm afternoon light streaming through a large window. She gazes thoughtfully at an open laptop, her expression serene and focused.",
    "creator_id": "user-1",
    "content_type": "image_description"
  }'
```
Response: `confidence: 0.67, attribution: uncertain, llm_score: 0.90`

**Example — Human image description:**
```bash
curl -X POST http://localhost:5001/submit \
  -H "Content-Type: application/json" \
  -d '{
    "text": "this is from my birthday last year, we were all crammed into my kitchen and someone knocked over the cake but we still ate it lol. the lighting is terrible but i love this photo.",
    "creator_id": "user-2",
    "content_type": "image_description"
  }'
```
Response: `confidence: 0.10, attribution: likely_human, llm_score: 0.00`

Invalid `content_type` values return a `400` with a clear error message. The `content_type` field is stored in every audit log entry.
